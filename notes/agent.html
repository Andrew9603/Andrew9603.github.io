Hermes Agent：一个会长期记事、会沉淀技能的 Agent 框架 ​

最近 Agent 框架实在太多了。刚开始看到 Hermes Agent 的时候，我的第一反应其实也比较普通：是不是又一个命令行版 AI 编程助手？看完官方文档和一些介绍后，我觉得它的重点不在"聊天更聪明"，而在"能不能长期作为一个常驻的数字同事来用"。

普通聊天助手更像一个问答窗口，一轮对话解决一件事。Hermes Agent 的设计目标要更往前走一步：它希望自己能记住项目背景，能把解决过的问题沉淀成技能，能通过消息平台长期在线，也能在沙箱里执行命令。换句话说，它不是只帮你"说"，而是尽量往"记住、执行、复用、持续改进"这个方向靠。

这篇文章不是那种"最强 Agent 框架来了"的宣传文。我更想按一个工科学生能看懂的方式，把 Hermes Agent 拆开讲清楚：它到底是什么，它和普通 Agent 框架差在哪，部署时要注意什么，以及它适合放在哪些真实场景里。

1. 先说结论：Hermes Agent 解决的不是"会不会聊天"，而是"能不能长期工作" ​

如果只把大模型当聊天机器人用，很多 Agent 框架看起来都差不多：输入任务，大模型规划步骤，调用工具，最后返回结果。这个流程本身并不稀奇。真正难的是任务结束之后怎么办。今天修过的 bug，明天还记不记得？这个项目的目录结构，下次还要不要重新解释？某个 API 的坑踩过一次，下一次能不能自动避开？

Hermes Agent 比较核心的想法就是：不要让每次任务都从零开始。它通过持久记忆保存关键事实，通过 Skill 保存可复用流程，通过会话搜索找回历史对话，再通过消息网关和定时任务让 Agent 不只停留在一个终端窗口里。官方文档把它描述成带有闭环学习能力的 Agent：能从经验里创建技能，使用中继续改进技能，并在跨会话过程中形成对用户和项目的长期理解。

从使用者角度看，这个设计有点像把"个人知识库 + 自动化脚本 + 命令行助手 + 多平台机器人"揉在了一起。它未必适合所有人，但如果你经常处理重复的开发、运维、数据分析、文档整理任务，这种"越用越熟"的思路是有价值的。

2. 为什么普通 Agent 框架容易用着用着就累？ ​

很多 Agent 项目刚开始体验都不错，因为 demo 任务一般比较短，工具也比较少。但到了真实项目里，问题会立刻变复杂。比如你让 Agent 帮你修一个后端接口，它需要知道项目结构、启动方式、测试命令、数据库配置、团队代码风格，还要知道之前修过哪些类似问题。如果这些东西每次都靠你重新贴给它，最后反而变成了"人类给 AI 打工"。

还有一种情况是重复任务很多。比如每周做一次代码审查，每天生成一次项目报告，隔一段时间排查一次部署失败。每次都让大模型临时规划，既浪费 token，也容易出现前后不一致。更工程化的做法应该是：第一次可以让 Agent 探索，第二次开始就应该沉淀流程，第三次就尽量复用稳定方案。

Hermes Agent 的设计正好围绕这些问题展开。它不是简单增加工具数量，而是把"记忆""技能""网关""调度""沙箱"放在同一个框架里，让 Agent 有机会从一次次任务里积累东西。

3. 整体架构：一条任务执行线，一条经验沉淀线 ​

理解 Hermes Agent 时，我建议先别急着看所有功能，而是把它拆成两条线。

第一条线是任务执行线。用户从 CLI、桌面端或者 Telegram、Slack、飞书这类消息平台发起任务，Hermes Agent 根据上下文和可用工具进行规划，然后调用终端、文件、MCP 工具、浏览器等能力完成任务，最后把结果返回给用户。

第二条线是经验沉淀线。任务执行过程中产生的环境事实、用户偏好、失败经验、可复用步骤，会被整理到 Memory 或 Skill 里。下次遇到类似任务时，Agent 不需要完全重新摸索，而是可以先读已有经验，再决定怎么做。

这两条线合起来，才是 Hermes Agent 和普通"一次性调用工具"的 Agent 最大区别。前者关心的是"这次任务怎么完成"，后者还关心"这次完成后能留下什么"。

4. 持久记忆：不是无限塞上下文，而是保存关键事实 ​

一听到"长期记忆"，很多人会误以为就是把所有历史聊天都塞进提示词。这样当然不现实，因为上下文长度和成本都会爆炸。Hermes Agent 的做法更克制：它把真正重要、应该常驻的东西放到有限长度的记忆文件里，把完整历史会话放到 SQLite 里，需要的时候再检索。

官方文档里提到，Hermes 的核心记忆由两个文件构成：MEMORY.md 和 USER.md。前者更偏 Agent 自己的工作笔记，比如机器环境、项目约定、踩坑经验；后者更偏用户画像，比如用户偏好的表达方式、常用工作流、沟通习惯。它们会在会话开始时作为固定快照注入系统提示词，中途即使更新，也要到下一次会话才会体现在提示词中。这个设计主要是为了让记忆稳定，同时避免每次中途改 prompt 影响性能缓存。

除此之外，Hermes 还会把会话历史存在 SQLite 中，并支持 FTS5 全文检索。这个设计很像我们自己做 RAG 时的思路：高频关键事实放在短记忆里，低频细节放在历史库里，用到时再查。这样既能记住事，又不至于让提示词越长越失控。

一个简单的记忆配置大概可以这样理解：

yaml
# ~/.hermes/config.yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200
  user_char_limit: 1375
  write_approval: true

这里我个人会把 write_approval 打开。原因很简单：Agent 自动保存记忆虽然方便，但它也可能把一次临时偏好误判成长期偏好。比如你今天说"这次回答短一点"，不代表你以后所有问题都要短。让记忆写入经过一次确认，麻烦一点，但更稳。

5. Skill 系统：把"做过一次的事"变成"下次可复用的方法" ​

如果说 Memory 存的是"事实"，那 Skill 存的更像"流程"。比如你让 Agent 修过一次 GitHub Actions 部署失败，它可以把排查路径整理成一个技能：先看失败日志，再判断依赖安装、环境变量、构建路径、权限配置，最后给出修复步骤。下次再遇到类似问题，它就不用完全从零开始。

Hermes 的技能文件采用 Markdown 形式，官方文档中也提到它兼容 agentskills.io 这种开放技能标准。这个设计挺有意思，因为 Markdown 本身足够简单，人能读，Agent 也能读，版本管理也方便。相比把所有逻辑写死在代码里，Skill 更像一种"软流程"：既可以给 Agent 指路，又保留了一定弹性。

一个很简化的 SKILL.md 可以长这样：

md
---
name: github-actions-debug
description: 排查 GitHub Actions 构建失败、部署失败、权限错误等问题
---

# GitHub Actions Debug Skill

## 适用场景

当用户贴出 GitHub Actions 失败日志，或者仓库部署失败时，使用这个技能。

## 排查步骤

1. 先定位失败发生在哪个 job 和 step。
2. 查看是否是依赖安装失败，例如 Node、Python、Java 版本不匹配。
3. 查看是否是路径问题，例如构建产物目录和部署配置不一致。
4. 查看是否是权限问题，例如 Pages、Actions、token 权限不足。
5. 给出最小修改方案，不要一上来大改工作流。

## 输出要求

- 先说明错误原因。
- 再给出要改的文件和代码片段。
- 最后说明如何重新触发构建。

这个例子不复杂，但它说明了 Skill 的意义：把一次任务中总结出的经验固定下来。这样 Agent 后面处理同类任务时，不只是"靠模型记忆"，而是有了明确的操作手册。

6. MCP：把外部工具接进来，但别把所有工具都暴露出去 ​

现在很多 Agent 框架都会提 MCP，也就是 Model Context Protocol。简单理解，MCP 就是给 Agent 接外部工具的一套协议。以前每个工具都要单独适配，现在可以通过 MCP server 把 GitHub、数据库、Figma、Sentry、Linear 等工具用相对统一的方式接进 Agent。

Hermes Agent 支持本地 stdio 类型的 MCP server，也支持远程 HTTP MCP server。它还能在启动时发现工具，并把这些工具注册进自己的工具系统里。这个能力很有用，因为 Agent 的边界不再局限于本地终端，而是可以扩展到很多外部系统。

不过这里要注意一个工程问题：工具不是越多越好。工具太多会导致模型选择困难，也会扩大安全风险。比较好的做法是给不同场景暴露不同的工具白名单。比如你只让它查 GitHub issue，就不要顺手给它仓库删除权限；你只让它查数据库，就不要给写库权限。

一个 MCP 配置例子可以写成这样：

yaml
# ~/.hermes/config.yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      allow:
        - search_issues
        - get_issue
        - list_pull_requests

如果是远程 HTTP 类型的 MCP server，可以是这种形式：

yaml
mcp_servers:
  internal_docs:
    url: "https://mcp.example.com/mcp"
    headers:
      Authorization: "Bearer ***"
    tools:
      allow:
        - search_docs
        - read_doc

这类配置的重点不是语法本身，而是权限边界。Agent 的能力越强，越要控制它能看到什么、能调用什么、能不能并行调用、能不能写入外部系统。

7. 消息网关：让 Agent 不只待在终端里 ​

Hermes Agent 还有一个比较突出的功能是消息网关。官方文档列出了很多接入入口，包括 Telegram、Discord、Slack、WhatsApp、Signal、飞书、钉钉、企业微信、邮件、SMS 等。对普通个人用户来说，这意味着你可以把它放在一台 VPS 或服务器上，然后通过聊天软件随时叫它干活。

这个点和普通命令行助手区别很明显。命令行助手通常跟着你的电脑走，电脑关了，它也就停了。消息网关的思路是让 Agent 常驻在线，任务可以从不同入口进来，但背后还是同一个 Agent、同一套记忆和配置。

比如你可以设想一个简单场景：早上在手机 Telegram 上让它检查昨晚的定时任务日志；中午在公司电脑终端里继续让它修代码；晚上让它把当天项目进度整理成一份报告。入口换了，但上下文和长期记忆不应该断。

启动网关的命令大概是这样：

bash
# 交互式配置消息网关
hermes gateway setup

# 前台启动，适合调试
hermes gateway

# Linux 上注册为后台服务
hermes gateway install
hermes gateway start
hermes gateway status

如果只是本地试用，这一步不是必须的。但如果你想把 Hermes Agent 当长期在线的个人助手，网关就是比较关键的一层。

8. 安全边界：Agent 能执行命令，就不能只谈"方便" ​

只要一个 Agent 能读文件、写文件、执行命令、联网访问，就必须认真看安全设计。因为它不再只是文本生成器，而是能对系统产生实际影响的自动化程序。Hermes 官方文档里专门写了安全部分，包括用户授权、危险命令审批、容器隔离、MCP 凭证过滤、上下文文件扫描、跨会话隔离和输入参数校验。

从个人部署角度，我觉得最重要的是三件事。

第一，不要一开始就开完全自动审批。比如删除文件、覆盖目录、修改系统配置、访问敏感路径等操作，最好让 Agent 先问你。方便和安全永远有个平衡，刚开始用的时候宁愿慢一点。

第二，尽量让执行环境跑在 Docker、Modal、Daytona 这类隔离环境里。官方文档也提到，Docker 后端会使用一系列安全加固参数，比如降低 Linux capabilities、限制进程数量、使用 tmpfs 等。对生产网关来说，沙箱比本地裸跑更稳。

第三，API Key 和内部系统凭证要最小暴露。能只给只读权限，就不要给写权限；能只暴露一个 MCP server 需要的环境变量，就不要把整个 shell 环境都传进去。

一个偏保守的安全配置思路可以这样写：

yaml
approvals:
  mode: manual
  timeout: 60
  cron_mode: deny

terminal:
  backend: docker
  docker_image: "nikolaik/python-nodejs:python3.11-nodejs20"
  docker_forward_env: []
  container_cpu: 1
  container_memory: 5120
  container_persistent: true

这个配置的含义很直接：危险命令需要人工确认，定时任务里遇到危险命令默认拒绝，执行命令放到 Docker 里，并且不默认把环境变量塞进容器。

9. 怎么快速跑起来？ ​

如果只是想体验，官方给出的方式比较简单。Linux、macOS、WSL2 或 Termux 可以直接用安装脚本：

bash
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash

安装后一般先跑配置向导：

bash
hermes setup
hermes model
hermes doctor

其中 hermes model 用来选择模型提供商和具体模型。Hermes 的一个特点是模型绑定没有那么死，可以接 Nous Portal、OpenRouter、OpenAI 兼容端点、本地 vLLM 等。对国内用户来说，如果使用 OpenAI 兼容 API 或者一些国产模型网关，也可以按自定义 endpoint 的方式接入，但要注意上下文长度和工具调用兼容性。

最简单的启动方式就是：

bash
hermes

如果想直接问一句，也可以：

bash
hermes chat -q "帮我列出当前项目的目录结构，并判断它是什么技术栈"

如果是 Windows 用户，建议优先走 WSL2。虽然现在有桌面安装方式，但很多开发、命令行、依赖安装、权限问题，在 Linux 环境下还是更容易处理。

10. 一个简单使用场景：让它沉淀项目排查经验 ​

假设你有一个前后端项目，经常遇到部署失败。第一次你可以直接让 Hermes 看日志：

bash
hermes chat -q "这是 GitHub Actions 的报错日志，请帮我判断失败原因，并给出最小修复方案" < error.log

如果它完成了排查，并且你发现这个流程以后还会反复用，就可以让它沉淀成技能：

text
请把这次 GitHub Actions 部署失败的排查流程整理成一个可复用 Skill。
要求包括：适用场景、排查顺序、常见错误、输出格式。

下次再遇到类似问题，你就不是重新问"这是什么错误"，而是让 Agent 带着已有 Skill 去处理。这个过程很像我们自己写项目笔记，只不过 Hermes 希望把"写笔记"和"下次调用笔记"也交给 Agent 自动完成。

11. 它适合哪些人？ ​

我觉得 Hermes Agent 比较适合三类人。

第一类是经常做开发和运维的人。比如代码审查、CI/CD 排错、服务器日志检查、定时备份、定期报告，这些任务都有比较明显的重复性，也适合沉淀技能。

第二类是有长期项目上下文的人。比如你长期维护一个开源项目，或者做一个研究课题，里面有很多目录结构、实验配置、论文笔记、数据处理脚本。如果 Agent 每次都能记住一点，下次就能少解释一点。

第三类是想折腾自托管 Agent 的人。Hermes 的消息网关、MCP、沙箱、技能系统都比较适合做个人自动化工作台。如果你只想网页里问两句，那它可能有点重。

12. 它不适合哪些情况？ ​

如果你只是偶尔写点代码，或者只需要一个简单聊天助手，Hermes Agent 可能会显得复杂。它涉及模型配置、密钥管理、执行环境、工具权限、消息平台接入，这些东西都需要一点工程基础。

如果你的任务高度敏感，比如涉及生产数据库写入、内部机密、财务数据，也不要一上来让 Agent 自动执行。应该先做只读接入、日志审计、审批流程和沙箱隔离。Agent 的自动化能力越强，越不能跳过安全设计。

另外，所谓"自我进化"也不要理解成它会自动训练出一个新模型。更准确地说，它主要是在技能、提示词、工具描述、流程策略这些文本层面做持续优化。这个方向很实用，但不是魔法。

13. 和普通 Agent 框架相比，我觉得它最大的不同 ​

我个人看下来，Hermes Agent 最值得关注的不是"支持多少模型"或者"接了多少平台"，这些当然也重要，但很多框架都可以堆功能。真正有区分度的是它把经验沉淀当作一等公民：Memory 负责关键事实，Session Search 负责历史查找，Skill 负责流程复用，Self-Evolution 负责进一步优化技能和提示词。

这套设计如果落到真实工作里，大概可以理解成：第一天它只是一个助手，第二周它开始像一个懂项目的人，时间再长一点，它可能会积累出一套你自己的工作流。当然这也取决于你怎么用、怎么审查、怎么配置权限。

14. 小结 ​

Hermes Agent 这类框架的出现，说明 Agent 的重点正在从"单次任务完成"转向"长期上下文积累"。以前我们更关心大模型能不能把当前问题答对，现在还要关心它能不能记住项目、复用流程、接入工具、安全执行，并且在真实使用中越跑越稳定。

从工程角度看，我觉得可以把 Hermes Agent 理解成一个长期在线的 Agent 工作台：前面是用户入口，中间是模型和工具编排，后面是记忆、技能、历史检索和安全沙箱。它不一定是所有人的最佳选择，但它确实把"Agent 怎么长期工作"这个问题讲得比较完整。

如果后面我要自己做一个类似项目，我会重点参考它的三点：第一，记忆要分层，不能乱塞上下文；第二，技能要文件化，方便人和 Agent 共同维护；第三，工具权限要最小化，不能为了自动化把系统安全全部让出去。
