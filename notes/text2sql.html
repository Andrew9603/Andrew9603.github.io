Text2SQL 从 Demo 到生产 ​

1. 前言：Text2SQL 为什么看起来简单，做起来却很难？ ​

Text2SQL 这个方向刚听起来很容易理解：用户说一句话，系统把它翻译成 SQL，然后去数据库里查结果。比如用户问"上个月华东地区销售额是多少"，系统生成一条 SELECT，再把结果展示出来。这个想法对业务系统很有吸引力，因为很多报表、临时查询、数据分析需求，本质上都不是高频标准接口。如果每个问题都让后端开发单独写接口，开发成本会非常高，业务方也等不起。

但是只要真正做过一个稍微复杂一点的业务库，就会发现 Text2SQL 不是"把表结构塞给大模型"这么简单。数据库里的字段名经常很抽象，业务问题里的词和数据库字段也不一定一一对应；表之间的关联关系可能藏在代码里，而不是数据库外键里；一些指标听起来很自然，比如 GMV、留存率、挂科率、动销率，但数据库里并不存在一张叫"GMV 表"或者"挂科表"的东西。更麻烦的是，模型生成的 SQL 可能语法正确、执行也能跑，但业务含义其实是错的。

所以本文想讨论的不是"怎么写一个能演示的 Text2SQL Demo"，而是从工程角度拆一下：一个 Text2SQL 系统怎么从最原始的 Prompt 方案，一步步演进到更接近生产可用的系统。为了讲得清楚，我把它拆成三层：第一层是 Schema Prompt，第二层是 RAG + Agent，第三层是业务工具化。最后再补一部分目前比较新的 Text2SQL 技术趋势，以及一些能直接放进项目里的代码片段。

2. 第一层：把表结构告诉模型，让它直接写 SQL ​

最容易想到的方案是：把数据库表结构、字段名、字段类型全部写进 Prompt，然后让大模型根据用户问题生成 SQL。这个方案非常适合做第一版 Demo，因为实现成本低，链路也直观。用户输入一句话，系统拼接 Prompt，调用模型，拿到 SQL，再执行。

在这一层里，Prompt 里至少要包含几类信息：表名、字段名、字段类型、字段注释、索引信息、表之间的关联关系。如果只是给出字段名和类型，模型很容易猜错字段含义。比如 status 到底表示订单状态、支付状态，还是课程发布状态？type 到底是用户类型还是商品类型？这些词对数据库设计者来说可能很清楚，但对模型来说并不稳定。

这里还有一个很容易被忽略的点：索引信息。很多 Text2SQL Demo 只关注 SQL 能不能执行，不关注 SQL 会不会把数据库拖死。比如模型在千万级订单表上写了一个没有索引的模糊查询，或者在几个大表之间做了不受限制的 Join，这种 SQL 在测试库里可能没问题，但在线上就是事故。因此，生产系统不能只检查语法，还要考虑执行代价。

下面是一个非常简化的 Schema 采集脚本，用 PostgreSQL 举例。它的作用是把表、字段、类型、注释和索引信息整理成后续可检索的元数据。

python
# code/schema_loader.py
from dataclasses import dataclass
from typing import List
import psycopg2

@dataclass
class ColumnInfo:
    table_name: str
    column_name: str
    data_type: str
    is_nullable: bool
    comment: str | None

@dataclass
class IndexInfo:
    table_name: str
    index_name: str
    index_def: str


def load_columns(conn) -> List[ColumnInfo]:
    sql = """
    SELECT
        c.table_name,
        c.column_name,
        c.data_type,
        c.is_nullable,
        pgd.description AS comment
    FROM information_schema.columns c
    LEFT JOIN pg_catalog.pg_statio_all_tables st
        ON c.table_schema = st.schemaname
       AND c.table_name = st.relname
    LEFT JOIN pg_catalog.pg_description pgd
        ON pgd.objoid = st.relid
       AND pgd.objsubid = c.ordinal_position
    WHERE c.table_schema = 'public'
    ORDER BY c.table_name, c.ordinal_position;
    """
    with conn.cursor() as cur:
        cur.execute(sql)
        rows = cur.fetchall()
    return [
        ColumnInfo(
            table_name=r[0],
            column_name=r[1],
            data_type=r[2],
            is_nullable=(r[3] == "YES"),
            comment=r[4],
        )
        for r in rows
    ]


def load_indexes(conn) -> List[IndexInfo]:
    sql = """
    SELECT tablename, indexname, indexdef
    FROM pg_indexes
    WHERE schemaname = 'public'
    ORDER BY tablename, indexname;
    """
    with conn.cursor() as cur:
        cur.execute(sql)
        rows = cur.fetchall()
    return [IndexInfo(table_name=r[0], index_name=r[1], index_def=r[2]) for r in rows]


def format_schema_for_prompt(columns: List[ColumnInfo], indexes: List[IndexInfo]) -> str:
    by_table: dict[str, list[ColumnInfo]] = {}
    for col in columns:
        by_table.setdefault(col.table_name, []).append(col)

    index_by_table: dict[str, list[IndexInfo]] = {}
    for idx in indexes:
        index_by_table.setdefault(idx.table_name, []).append(idx)

    parts = []
    for table, cols in by_table.items():
        parts.append(f"Table: {table}")
        for col in cols:
            comment = f" -- {col.comment}" if col.comment else ""
            parts.append(f"  - {col.column_name}: {col.data_type}{comment}")
        if table in index_by_table:
            parts.append("  Indexes:")
            for idx in index_by_table[table]:
                parts.append(f"    - {idx.index_name}: {idx.index_def}")
        parts.append("")
    return "\n".join(parts)

这套方案能解决一些简单问题，比如"统计学生总数""查询某个班级学生名单""统计某个课程的平均分"。但是一旦问题涉及多表关联、业务口径和隐含规则，它很快就会暴露缺陷。

3. 第一层的瓶颈：模型知道字段，不代表知道业务 ​

举一个教务系统的例子。用户问："今年计算机 2301 班学生在 C 语言程序设计这门课中，期末比期中提升超过 20% 的学生有哪些？"

这个问题看起来是一句话，但里面有很多业务含义。首先，"今年"可能对应某个学年或学期；其次，"提升超过 20%"需要确定计算公式；再其次，期中成绩和期末成绩可能在同一张成绩表里，需要自连接；最后，还要通过学生表、班级表、选课表、课程表、考试定义表、成绩表等多张表关联起来。只把字段名告诉模型，模型并不能稳定理解这些业务规则。

一个可能的 SQL 如下：

sql
SELECT
    s.student_id AS student_id,
    s.name AS student_name,
    mid_score.score AS midterm_score,
    final_score.score AS final_score,
    ROUND(
        ((final_score.score - mid_score.score) / NULLIF(mid_score.score, 0)) * 100,
        2
    ) AS growth_rate
FROM sys_student s
JOIN sys_admin_class c
    ON s.class_id = c.class_id
JOIN edu_enrollment e
    ON s.student_id = e.student_id
JOIN edu_course_offering co
    ON e.offering_id = co.offering_id
JOIN edu_course_catalog cc
    ON co.course_id = cc.course_id
JOIN edu_semester sem
    ON co.semester_id = sem.semester_id
JOIN edu_exam_definition mid_def
    ON co.offering_id = mid_def.offering_id
   AND mid_def.exam_name = '期中考试'
JOIN edu_exam_score mid_score
    ON mid_def.exam_def_id = mid_score.exam_def_id
   AND mid_score.student_id = s.student_id
JOIN edu_exam_definition final_def
    ON co.offering_id = final_def.offering_id
   AND final_def.exam_name = '期末考试'
JOIN edu_exam_score final_score
    ON final_def.exam_def_id = final_score.exam_def_id
   AND final_score.student_id = s.student_id
WHERE c.class_name = '计算机2301班'
  AND cc.course_name = 'C语言程序设计'
  AND sem.name = '2024-2025-1'
  AND mid_score.score > 0
  AND ((final_score.score - mid_score.score) / mid_score.score) > 0.20
ORDER BY growth_rate DESC;

这条 SQL 的难点不在语法，而在"业务知识"。模型必须知道哪些表能连起来，哪些字段表示课程，哪些字段表示考试类型，提升幅度怎么算，以及怎么避免除零。也就是说，Text2SQL 的核心问题并不是单纯的 SQL 语法生成，而是自然语言、业务概念和数据库结构之间的对齐，这通常被称为 Schema Linking。

4. 第二层：RAG + Agent，让模型按需查资料 ​

当数据库只有几张表时，把全量 Schema 放进 Prompt 还能接受。但真实企业系统经常有几十张、几百张表，字段可能上千甚至上万。把所有表结构、字段说明、业务文档、示例 SQL 都塞给模型，不仅 token 成本高，而且会引入大量无关信息。模型在长上下文里可能注意不到关键字段，反而更容易生成错误 SQL。

所以第二层的思路是：不要让模型一次性背完整个数据库，而是让它像人一样按需查资料。用户提出问题后，系统先做检索，找出相关表、字段、指标口径、Join 路径和历史示例 SQL，再把这些精简后的上下文交给 SQL 生成 Agent。

这里的知识库不只是表结构，还应该包括几类内容：

  1. Schema 文档：表名、字段名、字段类型、字段说明、索引、主外键关系。
  2. 业务术语词典：例如"客单价 = 销售额 / 订单数"，"挂科 = 成绩低于及格线"。
  3. Join Path：常见表之间怎么关联，优先使用什么字段。
  4. 示例 SQL：业务问题和 SQL 的配对样本，尤其是高频复杂问题。
  5. 权限规则：哪些表可查，哪些字段敏感，哪些用户角色不能访问。

下面给一个轻量版本的检索实现，用 BM25 做关键词召回。实际项目里可以把 BM25 和向量检索结合起来，先召回，再重排。

python
# code/schema_retriever.py
from dataclasses import dataclass
from rank_bm25 import BM25Okapi
import jieba

@dataclass
class KnowledgeChunk:
    chunk_id: str
    title: str
    content: str
    kind: str  # schema / metric / join_path / example_sql / policy


def tokenize(text: str) -> list[str]:
    return [w.strip().lower() for w in jieba.lcut(text) if w.strip()]


class SchemaRetriever:
    def __init__(self, chunks: list[KnowledgeChunk]):
        self.chunks = chunks
        self.corpus_tokens = [tokenize(c.title + "\n" + c.content) for c in chunks]
        self.bm25 = BM25Okapi(self.corpus_tokens)

    def search(self, question: str, top_k: int = 8) -> list[KnowledgeChunk]:
        query_tokens = tokenize(question)
        scores = self.bm25.get_scores(query_tokens)
        ranked = sorted(enumerate(scores), key=lambda x: x[1], reverse=True)
        return [self.chunks[i] for i, score in ranked[:top_k] if score > 0]


def build_context(chunks: list[KnowledgeChunk]) -> str:
    parts = []
    for c in chunks:
        parts.append(f"[{c.kind}] {c.title}\n{c.content}")
    return "\n\n".join(parts)

第二层通常还会拆 Agent。一个比较实用的拆法是：

  • Router / Planner Agent：判断用户问题属于查数、解释指标、生成报表还是无法回答。
  • Schema Linking Agent：从知识库里找相关表、字段、指标和样例。
  • SQL Writer Agent：根据精简上下文生成 SQL。
  • SQL Reviewer Agent：检查 SQL 是否越权、是否编造字段、是否缺少限制条件。
  • Executor：在受控环境里执行 SQL，并返回结果、错误信息和执行耗时。

这种拆法的好处是上下文更清楚，每个 Agent 的任务更单一。模型不需要同时记住"你是 SQL 专家""你是权限专家""你是业务专家""你是数据库优化专家"等一堆身份。拆开之后，每个环节都可以单独评测，也便于替换实现方式。

5. SQL 安全：不能相信模型生成的任何 SQL ​

只要模型生成 SQL，就一定要做安全校验。不要因为系统里有 System Prompt 写着"只能生成 SELECT"，就真的相信模型永远不会输出危险语句。模型可能被用户诱导，也可能误生成修改语句，还可能生成非常慢的查询。

一个常见做法是用 SQL 解析器做 AST 级别校验，而不是简单用字符串判断。下面是一个基于 SQLGlot 的简化 Guardrail：

python
# code/sql_guardrail.py
import sqlglot
from sqlglot import exp

FORBIDDEN = (
    exp.Insert,
    exp.Update,
    exp.Delete,
    exp.Drop,
    exp.Alter,
    exp.TruncateTable,
    exp.Create,
)

class SQLValidationError(Exception):
    pass


def validate_readonly_sql(sql: str, allowed_tables: set[str], max_limit: int = 1000) -> str:
    try:
        parsed = sqlglot.parse_one(sql, dialect="postgres")
    except Exception as e:
        raise SQLValidationError(f"SQL parse failed: {e}")

    if not isinstance(parsed, exp.Select):
        raise SQLValidationError("Only SELECT statement is allowed")

    for node in parsed.walk():
        if isinstance(node, FORBIDDEN):
            raise SQLValidationError(f"Forbidden SQL operation: {type(node).__name__}")

    used_tables = {t.name for t in parsed.find_all(exp.Table)}
    illegal_tables = used_tables - allowed_tables
    if illegal_tables:
        raise SQLValidationError(f"Illegal table access: {illegal_tables}")

    limit = parsed.args.get("limit")
    if limit is None:
        parsed = parsed.limit(max_limit)
    else:
        try:
            n = int(limit.expression.name)
            if n > max_limit:
                parsed = parsed.limit(max_limit)
        except Exception:
            parsed = parsed.limit(max_limit)

    return parsed.sql(dialect="postgres")

这段代码只做了最基础的几件事：只允许 SELECT，禁止写操作，限制可访问表，自动补 LIMIT。真实项目还要继续加权限控制、字段脱敏、行级权限、慢查询超时、执行计划检查和审计日志。

6. 第三层：把高频复杂查询封装成业务工具 ​

第二层已经比第一层可靠很多，但它还有一个问题：成本高。每次复杂查询都要检索、规划、生成、执行、纠错，多轮 Agent 跑下来，耗时和 token 成本都不低。如果系统面对的是老板偶尔问几个问题，这个成本可能能接受；但如果是客服、运营、BI 看板这类高频场景，就会很吃力。

第三层的思路是：不是所有问题都要让模型临时写 SQL。对于高频、复杂、稳定的业务问题，可以把它沉淀成工具。模型负责理解用户意图和填写参数，真正的 SQL 和业务逻辑由后端工具执行。

比如"查询某班某门课程成绩提升超过一定比例的学生"这个问题，就可以封装成一个工具：

python
# code/tools.py
from pydantic import BaseModel, Field
from typing import Literal

class ScoreGrowthQuery(BaseModel):
    class_name: str = Field(..., description="班级名称，例如：计算机2301班")
    course_name: str = Field(..., description="课程名称，例如：C语言程序设计")
    semester_name: str = Field(..., description="学期名称，例如：2024-2025-1")
    previous_exam: str = Field("期中考试", description="对比基准考试")
    current_exam: str = Field("期末考试", description="当前考试")
    min_growth_rate: float = Field(0.2, ge=0, le=10, description="最低提升比例，0.2 表示 20%")
    order: Literal["asc", "desc"] = "desc"
    limit: int = Field(100, ge=1, le=500)


def query_score_growth(db, args: ScoreGrowthQuery):
    sql = """
    SELECT
        s.student_id,
        s.name AS student_name,
        mid_score.score AS previous_score,
        final_score.score AS current_score,
        ROUND(
            ((final_score.score - mid_score.score) / NULLIF(mid_score.score, 0)) * 100,
            2
        ) AS growth_percent
    FROM sys_student s
    JOIN sys_admin_class c ON s.class_id = c.class_id
    JOIN edu_enrollment e ON s.student_id = e.student_id
    JOIN edu_course_offering co ON e.offering_id = co.offering_id
    JOIN edu_course_catalog cc ON co.course_id = cc.course_id
    JOIN edu_semester sem ON co.semester_id = sem.semester_id
    JOIN edu_exam_definition mid_def
      ON co.offering_id = mid_def.offering_id
     AND mid_def.exam_name = %(previous_exam)s
    JOIN edu_exam_score mid_score
      ON mid_def.exam_def_id = mid_score.exam_def_id
     AND mid_score.student_id = s.student_id
    JOIN edu_exam_definition final_def
      ON co.offering_id = final_def.offering_id
     AND final_def.exam_name = %(current_exam)s
    JOIN edu_exam_score final_score
      ON final_def.exam_def_id = final_score.exam_def_id
     AND final_score.student_id = s.student_id
    WHERE c.class_name = %(class_name)s
      AND cc.course_name = %(course_name)s
      AND sem.name = %(semester_name)s
      AND mid_score.score > 0
      AND ((final_score.score - mid_score.score) / mid_score.score) > %(min_growth_rate)s
    ORDER BY growth_percent DESC
    LIMIT %(limit)s;
    """
    params = args.model_dump()
    with db.cursor() as cur:
        cur.execute(sql, params)
        columns = [desc[0] for desc in cur.description]
        rows = cur.fetchall()
    return [dict(zip(columns, row)) for row in rows]

这个工具的价值在于，它把复杂 Join、计算公式、除零保护、Limit、参数类型校验都固化在后端了。模型不用每次重新写一大段 SQL，只需要输出结构化参数：

json
{
  "tool": "query_score_growth",
  "arguments": {
    "class_name": "计算机2301班",
    "course_name": "C语言程序设计",
    "semester_name": "2024-2025-1",
    "previous_exam": "期中考试",
    "current_exam": "期末考试",
    "min_growth_rate": 0.2,
    "limit": 100
  }
}

这就是"让模型做它擅长的事，让程序做确定性的事"。模型擅长理解自然语言、抽取参数、在工具之间做选择；后端擅长执行确定性逻辑、安全控制和性能优化。

7. 工具怎么设计：少工具、多参数，不要工具满天飞 ​

工具化不是说每个查询都写成一个工具。如果工具太多，模型选择工具本身就会变困难，而且维护成本会爆炸。一个比较朴素但有效的原则是：少工具、多参数。

比如不要设计一堆过细的工具：

  • query_monday_schedule
  • query_tuesday_schedule
  • query_class_schedule
  • query_teacher_schedule
  • query_valid_schedule
  • query_future_schedule

更好的方式是设计一个参数比较完整的 query_schedule：

python
class ScheduleQuery(BaseModel):
    class_names: list[str] | None = None
    teacher_names: list[str] | None = None
    weekdays: list[int] | None = None
    semester_name: str
    status: list[Literal["draft", "valid", "expired"]] = ["valid"]
    start_week: int | None = None
    end_week: int | None = None

这样，用户问"查计算机 2301 班周一的正式课表"和"查张老师本学期第 3 到第 8 周的课表"，都可以落到同一个工具上。工具数量少了，模型选工具更容易；参数多了，表达能力仍然足够。

不过参数也不能无限增加。如果一个工具的参数复杂到模型经常填错，说明这个工具边界可能设计得不好。此时可以把它拆成两个工具，或者先通过 Planner Agent 做问题拆解。

8. "有形查询"和"无形指标"：生产系统真正难的是后者 ​

有形查询比较好理解，因为数据库里确实有对应的表。比如查学生、查课程、查订单、查商品、查库存。模型只要找对表和字段，基本就能生成查询。

无形指标就麻烦很多。比如电商里的 GMV、ERP 里的动销率、运营里的留存率、搜索系统里的 CTR、教务系统里的挂科率。这些概念在业务里很常见，但数据库里通常没有一张直接对应的表。它们是由多张表、多个过滤条件和计算公式组合出来的。

因此，在企业级 Text2SQL 系统里，指标口径文档非常重要。最好把指标也当成知识库的一部分，并且用结构化格式维护：

yaml
# metrics/gmv.yaml
name: GMV
description: 用户实际支付成功的商品成交总额
formula: SUM(order_item.pay_amount)
tables:
  - orders
  - order_items
filters:
  - orders.pay_status = 'PAID'
  - orders.is_deleted = false
time_field: orders.pay_time
common_dimensions:
  - region
  - channel
  - category
notes:
  - 不包含未支付订单
  - 退款是否扣减取决于报表口径，默认不扣减

生成 SQL 时，系统可以先检索到这个指标定义，再让模型根据用户问题补充时间、地区、渠道等维度。如果用户问"上个月华东地区 GMV"，模型就不需要猜 GMV 怎么算，而是按照系统维护的指标口径生成查询。

9. 低频问题怎么处理：不要一刀切 ​

工具化解决的是高频复杂问题，但系统总会遇到工具覆盖不到的问题。低频问题可以分成两类。

第一类是低频但简单的问题，比如"查某张小表里最近十条记录"。这种问题可以允许模型走原生 Text2SQL，但必须经过 Guardrail 校验。

第二类是低频且复杂的问题，比如跨很多表、涉及特殊业务口径、执行路径很长的问题。这种问题不建议直接信模型。更合理的做法是把它记录下来，进入人工审核或离线分析流程。如果这类问题出现次数越来越多，就说明它不再是真正的低频问题，可以把它沉淀成新的工具或新的指标口径。

这也是我觉得 Text2SQL 工程里最重要的闭环：不是一次性把系统做完，而是让系统能从线上问题中成长。每次失败样本都应该被记录，包括用户原问题、检索到的上下文、生成 SQL、执行结果、错误类型、耗时和人工修正版本。后续可以用这些样本更新知识库、补充示例 SQL、优化工具参数，甚至构造评测集。

10. 一个最小可用的 FastAPI 接口 ​

下面给一个简化版接口，把上面几个模块串起来。真实项目里还需要加入用户鉴权、数据库连接池、日志、超时控制和结果脱敏。

python
# code/app.py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI(title="Text2SQL Demo")

class QueryRequest(BaseModel):
    question: str
    user_role: str = "analyst"

class QueryResponse(BaseModel):
    sql: str | None = None
    answer: str | None = None
    rows: list[dict] | None = None
    route: str


@app.post("/query", response_model=QueryResponse)
def query(req: QueryRequest):
    # 1. 先判断是否命中业务工具
    tool_call = route_to_tool(req.question)
    if tool_call is not None:
        result = execute_tool(tool_call)
        return QueryResponse(route="tool", rows=result, answer="已通过业务工具查询完成")

    # 2. 未命中工具，则走 RAG + SQL 生成
    chunks = retriever.search(req.question, top_k=8)
    context = build_context(chunks)

    raw_sql = llm_generate_sql(question=req.question, context=context)

    # 3. SQL 安全校验
    allowed_tables = get_allowed_tables(req.user_role)
    try:
        safe_sql = validate_readonly_sql(raw_sql, allowed_tables=allowed_tables)
    except Exception as e:
        raise HTTPException(status_code=400, detail=str(e))

    # 4. 受控执行
    rows = execute_readonly_sql(safe_sql, timeout_seconds=5)

    # 5. 结果解释
    answer = llm_explain_result(req.question, safe_sql, rows)
    return QueryResponse(route="sql", sql=safe_sql, rows=rows, answer=answer)

这段接口体现了一个基本原则：优先走稳定工具，工具不覆盖时再走 SQL 生成。如果系统一开始工具很少，可以先以 RAG + SQL 生成作为主链路；随着日志积累，再逐步把高频复杂问题沉淀成工具。

11. 最新 Text2SQL 技术趋势：不是只靠一个大模型硬怼 ​

最近的 Text2SQL 研究和工程实践有几个明显方向。

第一个方向是任务分解。DIN-SQL 这类方法把 Text2SQL 拆成 Schema Linking、问题分类、SQL 生成和自我修正等步骤，说明复杂 SQL 生成不是一次 Prompt 就能稳定解决的。这个思路和工程上的多 Agent 拆分是相通的：把一个大问题拆成多个小问题，分别处理，再合成结果。

第二个方向是候选生成与选择。CHASE-SQL 采用多路径推理，让不同生成器产出多个 SQL 候选，再通过选择器挑出更好的结果。这说明在复杂场景下，单次生成不一定可靠，多候选 + 评审有时比"一次生成一步到位"更稳。

第三个方向是面向大规模 Schema 的检索增强。企业数据库往往有成百上千张表，甚至上万列。Spider 2.0 这类新基准已经开始强调真实企业工作流，包括云数据库、多 SQL 方言和复杂数据分析任务。BIRD 也强调 SQL 不仅要正确，还要考虑执行效率。因此，未来 Text2SQL 系统不能只追求语法正确，还要关注 Schema 检索、执行效率、权限边界和复杂工作流。

第四个方向是工具调用和 Agent 化。从工程角度看，Text2SQL 不应该只是"自然语言转 SQL"，而应该是一个数据分析 Agent。它需要理解问题、找表、查文档、调用工具、生成 SQL、校验 SQL、执行查询、解释结果，并把失败样本沉淀回系统。也就是说，Text2SQL 更像一个完整的工程系统，而不是一个孤立模型能力。

12. 怎么评测一个 Text2SQL 系统？ ​

如果只用"SQL 能不能执行"来评测，指标是不够的。一个 SQL 能执行，不代表它回答了用户问题。建议至少看下面几类指标：

  1. 可执行率：生成的 SQL 是否能被数据库执行。
  2. 执行正确率：结果是否和标准答案一致。
  3. 字段命中率：是否使用了正确的表和字段。
  4. 业务口径正确率：指标公式、过滤条件、时间窗口是否正确。
  5. 安全通过率：是否出现越权表、敏感字段、写操作、慢查询。
  6. 响应耗时：从用户提问到返回结果需要多久。
  7. Token 成本：每次查询消耗多少输入输出 token。
  8. 人工修正率：有多少问题最终需要人工介入。

可以先写一个简单评测脚本：

python
# code/evaluator.py
from dataclasses import dataclass

@dataclass
class EvalCase:
    question: str
    gold_sql: str
    gold_tables: set[str]
    gold_columns: set[str]

@dataclass
class EvalResult:
    executable: bool
    table_recall: float
    column_recall: float
    latency_ms: int
    error: str | None = None


def recall(pred: set[str], gold: set[str]) -> float:
    if not gold:
        return 1.0
    return len(pred & gold) / len(gold)


def evaluate_case(case: EvalCase) -> EvalResult:
    start = now_ms()
    try:
        pred_sql = generate_sql(case.question)
        safe_sql = validate_readonly_sql(pred_sql, allowed_tables=get_all_tables())
        execute_readonly_sql(safe_sql)
        executable = True
        error = None
    except Exception as e:
        pred_sql = ""
        executable = False
        error = str(e)

    pred_tables = extract_tables(pred_sql)
    pred_columns = extract_columns(pred_sql)
    return EvalResult(
        executable=executable,
        table_recall=recall(pred_tables, case.gold_tables),
        column_recall=recall(pred_columns, case.gold_columns),
        latency_ms=now_ms() - start,
        error=error,
    )

13. 总结：Text2SQL 的重点不是炫技，而是取舍 ​

如果只是做课程项目或者博客 Demo，第一层 Schema Prompt 就够用了。它实现快，也容易展示效果。但如果要做一个比较像样的工程项目，至少应该做到第二层：有知识库、有检索、有 SQL 校验、有执行反馈。否则一旦表多起来，模型很容易乱写字段、乱 Join、乱套业务口径。

如果目标是生产可用，第三层业务工具化基本绕不开。因为真实业务系统里，高频复杂问题往往有固定口径和固定流程。与其让模型每次临时生成一条风险很高的 SQL，不如把稳定逻辑沉淀成工具，让模型负责选择工具和填参数。这样系统的准确率、性能、安全性和可维护性都会更好。

我现在对 Text2SQL 的理解可以总结成一句话：Text2SQL 不是让大模型替后端写 SQL，而是把自然语言入口、业务知识库、SQL 生成、权限安全、工具调用和持续评测组合成一个可迭代的数据查询系统。

对普通开发者来说，比较现实的路线是：先做 Schema Prompt 跑通 Demo，再加 RAG 和 Guardrail 提升可用性，最后根据日志把高频复杂问题沉淀成业务工具。这样既不会一开始就把系统设计得过重，也不会停留在一个只能演示、不能落地的玩具系统。
