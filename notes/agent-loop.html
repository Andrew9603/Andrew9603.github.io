Agent Loop：从"写 Prompt"到"写 Loop"，AI 编程真正变了什么 ​

最近我看了几段 Boris Cherny 的采访。Boris 是 Anthropic 里 Claude Code 的创建者和负责人，所以他说 AI 编程的时候，不是站在旁边点评，而是站在产品和工程实践的一线讲自己的工作方式。

这几天最出圈的一句话，大概是：

I don't prompt Claude anymore. I have loops running that prompt Claude and figure out what to do. My job is to write loops.

我把它翻得口语一点，就是：

我已经不写 prompt 了，我写 loop。

这句话听着很像一句互联网金句，但它背后不是"Prompt 过时了"这么简单。更准确地说，是 AI 编程正在从"人给模型写一次提示词"转向"人设计一个能持续运行的任务闭环"。

以前我们用 AI 编程，更多是在写 Prompt：

text
帮我写一个接口。
帮我解释这个报错。
帮我改一下这段代码。
帮我生成一个 SQL。

现在一些更前沿的用法已经变成了：

text
给 Agent 一个目标。
让它自己读代码、改代码、运行测试、看报错、继续修。
直到任务完成，或者碰到边界后停下来。

这就是从 Prompt 到 Loop 的变化。

这篇文章我会尽量结合真实事件来讲，不只讲概念。我们先看 Boris 这句话是在什么背景下出现的，再看 Agent Loop 到底是什么，最后再给一个简化代码例子，帮助理解它在工程里怎么落地。

1. 这句话不是凭空冒出来的：先看几个时间点 ​

如果只截一句"我写 loop"，很容易把它理解成营销话术。但把 Boris 过去几个月的几次采访串起来看，会发现这其实是一条比较清晰的线。

1.1 2026 年 2 月：YC / Lightcone 和 Lenny's Podcast 里，Boris 开始讲"coding is solved" ​

2026 年 2 月，Boris 出现在 Y Combinator 的 Lightcone 节目里，主题是 Claude Code 的诞生和 AI 编程的变化。这期节目在 Apple Podcasts 上的标题是 Inside Claude Code With Its Creator Boris Cherny，发布时间是 2026 年 2 月 17 日，时长大约 50 分钟。

差不多同一时期，他也上了 Lenny Rachitsky 的访谈，标题是 What happens after coding is solved。Lenny 这期访谈的介绍里列了很多关键词：Claude Code 如何从一个 quick hack 发展起来、为什么 Boris 认为 coding 已经被解决、为什么"少配人 + 给足 token"会逼出更多自动化，以及 Boris 给团队成员强调的几个原则。

Business Insider 2 月 18 日也报道了 YC Lightcone 里的内容。报道里提到，Boris 认为"coding is practically solved"，未来传统意义上的 "software engineer" 这个岗位称呼可能会开始变化，很多人会更像 builder 或 product manager。

这部分先不讨论他说得是不是太激进。至少可以确定一件事：Boris 从 2 月开始就在公开表达一个方向——AI 编程的重点，正在从"人手写代码"转向"人设定目标、AI 执行、人负责验证和判断"。

1.2 2026 年 5 月：Sequoia AI Ascent 上，他讲到"几千个 agent 晚上干活" ​

5 月 4 日，Boris 在 Sequoia Capital 的 AI Ascent 访谈里进一步讲了自己的实际工作方式。Business Insider 5 月 13 日报道了这场访谈：Boris 说自己经常在手机上开着 5 到 10 个 Claude Code sessions，每个 session 里又包含多个 agent；晚上通常会有"几千个"agent 做更深的工作。

这里有一个很关键的点：他不只是说"我让 Claude 写代码"，而是提到了 Claude Code 里用于持续自动化的能力，比如 /loops 和 Routines。报道中说，/loops 可以本地通过 cron 定时运行，而 Routines 可以跑在服务器上，让任务在电脑合上以后也能继续执行。

这个场景其实已经很接近 Agent Loop 的核心形态了：不是每次等用户发一条 Prompt，而是让 agent 在一个规则明确的循环里持续工作。

1.3 2026 年 6 月：Acquired Unplugged / WorkOS 访谈里，"我写 loop"被大量传播 ​

6 月 2 日，WorkOS 发布了一篇对 Boris 在 Acquired Unplugged presented by WorkOS 访谈的总结。这场访谈由 Acquired 的 Ben Gilbert 和 David Rosenthal 主持。

WorkOS 的总结提到，Claude Code 最早在 Anthropic 的一个 prototyping team 里起步。刚开始它可能只写 Boris 10% 到 20% 的代码，后来随着 Sonnet 3.5、Sonnet 4、Opus 等模型能力提升，Claude Code 参与写代码的比例快速上升。Boris 还提到自己因为一个月没打开 IDE，干脆把 IDE 卸载了。

更重要的是，WorkOS 把这件事概括成一个阶段变化：Boris 的工作从"写代码"变成了"编排 agent"。也就是那句非常容易传播的话：他现在不是直接 prompt Claude，而是在写 loops，让 loops 去 prompt Claude、判断下一步做什么。

所以，Agent Loop 不是一个突然冒出来的新词。它背后至少有这条事件线：

text
2026.02：YC / Lenny 访谈，Boris 公开谈 coding is solved 和工程角色变化
2026.05：Sequoia AI Ascent，Boris 谈多个 sessions、几千个 agents、/loops 和 Routines
2026.06：Acquired Unplugged / WorkOS，Boris 讲"我不写 prompt，我写 loop"

这条线串起来看，意思就比较清楚了：AI 编程不是简单从"人写代码"变成"AI 写代码"，而是从"人亲自执行每一步"变成"人设计执行系统"。

2. 为什么只写 Prompt 不够了？ ​

如果任务很短，Prompt 当然够用。

比如你让模型写一个函数：

text
帮我写一个 Python 函数，把字符串里的中文标点替换成英文标点。

这种任务没有太多状态，也不需要复杂验证。模型一次生成，最多你再改一下就行。

但真实开发里的很多任务不是这样。比如你让 Agent 修一个后端测试失败的问题，它可能需要做这些事：

  1. 读取报错日志；
  2. 找到失败的测试；
  3. 搜索相关代码；
  4. 判断是不是接口返回值错了；
  5. 修改代码；
  6. 重新跑测试；
  7. 如果还失败，就继续看新报错；
  8. 最后确认测试通过。

这就不是一句 Prompt 能稳定覆盖的问题。因为每一步都会产生新的状态，下一步又依赖上一步的结果。

如果用普通 Prompt 做这类任务，你会发现自己在不断给 AI "接力"：

text
你：这是报错日志，你看看。
AI：可能是这里的问题。
你：我跑了测试，还是失败，这是新日志。
AI：那可能是另一个文件。
你：你忘了前面已经改过这个函数。
AI：哦，那我重新分析一下。

这样用久了，人反而变成了 AI 的状态管理器。

Loop 的意义，就是把这些人工接力的步骤系统化。系统自己保存状态，自己读取反馈，自己决定下一轮要不要继续。

3. Agent Loop 到底是什么？ ​

我自己的定义很简单：

Agent Loop 就是让大模型在"观察环境 → 推理计划 → 执行动作 → 读取结果 → 验证反馈 → 决定下一步"中循环运行的机制。

Oracle 开发者博客在解释 AI Agent Loop 时，也把它看成 chatbot 和 agent 的关键差异：chatbot 通常是单次响应，而 agent 会在工具调用和环境反馈中持续迭代，直到任务完成或达到停止条件。

一个基础 Loop 大概是这样：

text
目标输入
  ↓
读取上下文
  ↓
模型规划下一步
  ↓
调用工具执行
  ↓
观察执行结果
  ↓
验证是否成功
  ↓
成功：停止
失败：带着反馈进入下一轮

这里面有几个关键词。

3.1 观察环境 ​

Agent 不能只靠模型"脑补"。它要能看见当前环境。

比如：

  • 当前项目目录是什么；
  • 测试报错在哪里；
  • Git diff 改了哪些文件；
  • 数据库返回了什么结果；
  • 外部 API 是否调用成功；
  • 用户上一次反馈是什么。

没有观察，就没有真正的闭环。

3.2 执行动作 ​

Agent 不只是输出文字，它要能行动。

行动可以是：

  • 读文件；
  • 改文件；
  • 运行 shell 命令；
  • 调用 Python；
  • 查询数据库；
  • 访问 API；
  • 创建 PR；
  • 写文档。

这也是 Agent 和普通聊天最大的区别。普通聊天只回答，Agent 会改变环境。

3.3 验证反馈 ​

Loop 的关键不是多跑几轮，而是每一轮都能验证。

代码任务可以跑测试；SQL 任务可以执行 SQL；接口任务可以看返回值；文档任务可以检查格式和引用；运维任务可以看服务状态。

如果没有验证，Loop 就只是"反复生成"。反复生成不等于变好，甚至可能越改越乱。

3.4 停止条件 ​

Loop 不是无限循环。一个靠谱的 Loop 必须知道什么时候停。

常见停止条件包括：

  • 任务完成；
  • 测试通过；
  • 达到最大步数；
  • 超过 token 或时间预算；
  • 连续失败次数太多；
  • 触发危险操作，需要人工审批；
  • 用户主动中止。

没有停止条件的 Agent Loop，在真实工程里非常危险。

4. 从 Prompt 工程到 Loop 工程，开发者的位置变了 ​

Prompt 工程更关心一句话怎么写。比如：

  • 角色设定怎么写；
  • 输出格式怎么限制；
  • 示例怎么放；
  • 约束条件怎么表达；
  • 要不要让模型先分析再回答。

这些东西仍然重要，但在 Agent 场景里，它们已经不是全部。

Loop 工程更关心系统怎么运行：

  • 任务如何拆成多步；
  • 状态如何保存；
  • 工具如何选择；
  • 哪些动作允许自动执行；
  • 哪些动作必须人工确认；
  • 如何验证结果；
  • 失败后如何重试；
  • 什么时候停止；
  • 怎么记录整个过程。

一句话概括就是：

Prompt 更像一次请求，Loop 更像一套可迭代系统。

这也是为什么很多 Agent 框架都绕不开状态机、工具调用、检查点、日志、权限、评估这些东西。因为一旦 Agent 开始"做事"，它就不只是一个聊天窗口，而是一个会影响外部环境的自动化系统。

5. Boris 的几个真实案例说明了什么？ ​

5.1 卸载 IDE：入口变了，不代表工程不重要了 ​

Boris 说自己因为很久没打开 IDE，所以卸载了 IDE。这个说法很夸张，也很容易被误读。

我觉得更准确的理解是：他的主要开发入口变了。

过去是：

text
IDE → 人写代码 → 人运行测试 → 人修 bug → 人提交

现在变成：

text
Claude Code / Agent → 读代码 → 改代码 → 跑测试 → 人审查和给方向

也就是说，不是工程不重要了，而是人从"亲自敲每一行代码"变成了"设定目标、约束边界、审核结果"。

这对普通开发者也有参考意义。我们不一定要卸载 IDE，但可以开始思考：哪些重复开发动作可以交给 Agent Loop？哪些地方必须保留人工判断？

5.2 几千个 Agent 过夜：AI 编程开始走向异步和并行 ​

Sequoia 访谈里，Boris 说晚上会有大量 agent 做 deeper work。这件事很有代表性。

它说明 AI 编程不再只是一个聊天窗口，而开始接近异步任务系统。

比如：

text
Agent A：看 GitHub issues
Agent B：查用户反馈
Agent C：跑测试
Agent D：重构某个模块
Agent E：整理文档
Agent F：生成方案比较

第二天人再看结果，筛选能用的部分，继续推进。

这里的关键不是"数量越多越好"，而是任务可以被拆分、并行、异步执行。这个思路和传统软件里的任务队列、后台 job、cron、CI/CD 很像。

5.3 /loops 和 Routines：Loop 开始产品化 ​

如果只是我们自己写一个 while 循环，那还只是工程实验。关键在于 Claude Code 已经开始把 Loop 变成产品能力。

从报道里看，/loops 和 Routines 这类功能的意义在于：

  • 任务可以定时运行；
  • 可以在后台运行；
  • 可以在服务器上继续运行；
  • 不一定依赖用户一直盯着屏幕；
  • 可以把重复工作沉淀成持续流程。

这和普通 Prompt 最大的区别是：Prompt 是人来触发，Loop 是系统按规则推进。

6. 一个简单代码例子：Loop 骨架长什么样？ ​

下面给一个极简版代码，模拟一个"自动修复测试失败"的 Loop。它不是为了直接上线，而是用来理解 Agent Loop 的骨架。

python
from dataclasses import dataclass, field
from typing import List

@dataclass
class LoopState:
    goal: str
    step: int = 0
    done: bool = False
    failures: int = 0
    history: List[str] = field(default_factory=list)
    last_action: str = ""
    last_observation: str = ""


def plan_next_action(state: LoopState) -> str:
    if state.step == 0:
        return "run_tests"

    if "AssertionError" in state.last_observation:
        return "inspect_failure"

    if "root cause found" in state.last_observation:
        return "patch_code"

    return "run_tests"


def execute(action: str) -> str:
    fake_env = {
        "run_tests": "FAILED: AssertionError in test_api.py",
        "inspect_failure": "root cause found: API returns 500 instead of 200",
        "patch_code": "patched app/api.py",
    }
    return fake_env.get(action, "unknown action")


def validate(observation: str) -> bool:
    return "all tests passed" in observation.lower()


def should_stop(state: LoopState, max_steps=8, max_failures=3) -> bool:
    if state.done:
        return True
    if state.step >= max_steps:
        return True
    if state.failures >= max_failures:
        return True
    return False


def run_loop(goal: str):
    state = LoopState(goal=goal)

    while not should_stop(state):
        action = plan_next_action(state)
        observation = execute(action)

        state.step += 1
        state.last_action = action
        state.last_observation = observation
        state.history.append(
            f"step={state.step}, action={action}, observation={observation}"
        )

        if action == "patch_code":
            # 真实系统里这里会重新运行测试
            state.last_observation = "all tests passed"
            state.done = True
            break

        if "FAILED" in observation:
            state.failures += 1

    return state


result = run_loop("修复 pytest 失败")
print(result.done)
print("\n".join(result.history))

这个例子虽然简单，但已经包含了几个关键模块：

  • goal：任务目标；
  • state：当前状态；
  • plan_next_action：根据状态规划下一步；
  • execute：执行动作；
  • validate：验证是否成功；
  • should_stop：停止条件；
  • history：过程日志。

真实系统会把这些模块换成真正的大模型调用、文件系统、shell 工具、测试框架、数据库和审计日志。但基本骨架是一样的。

7. 真实落地时，Loop 最容易踩的坑 ​

7.1 成本失控 ​

Agent Loop 一定比单轮问答贵。因为它会多轮调用模型、多轮调用工具、多次保存上下文。如果再加上多个 agent 并行，成本会继续上升。

所以要加边界：

text
最大轮数
最大运行时间
最大 token 预算
最大失败次数
工具调用次数上限

否则 Loop 可能跑得很开心，账单也跑得很开心。

7.2 没有验证，只是在重复生成 ​

很多人做 Agent demo，最大的问题是没有验证器。模型说自己完成了，系统就认为完成了。

这不可靠。

更好的做法是：

text
代码任务 → 跑测试
SQL 任务 → 执行 SQL / SQLGlot 校验
接口任务 → 看 HTTP 状态码和响应结构
文档任务 → 检查引用、格式、缺失字段
运维任务 → 看服务状态和日志

Loop 的核心价值是反馈，不是循环次数。

7.3 权限放太大 ​

Agent 能执行命令，就一定要考虑安全边界。

比如这些动作最好不要默认放开：

  • 删除文件；
  • 覆盖配置；
  • 写生产数据库；
  • 调用付费 API；
  • 部署生产环境；
  • 读取敏感密钥。

比较稳的策略是：

text
默认只读
危险动作人工审批
工具按场景白名单开放
所有动作写日志
生产环境单独隔离

7.4 状态越来越乱 ​

Loop 跑多了以后，history 会越来越长，工具输出会越来越多，模型很容易丢重点。

所以真实系统通常还需要：

  • 状态压缩；
  • 关键事实提取；
  • checkpoint；
  • 任务分阶段；
  • 失败摘要；
  • 可回滚的 diff。

这也是为什么 Agent Loop 更像系统工程，而不是 Prompt 技巧。

8. 普通开发者可以从哪里开始？ ​

我觉得不用一上来就做一个"几千个 agent 晚上自动干活"的系统。普通开发者更适合从小 Loop 开始。

8.1 GitHub Actions 排错 Loop ​

text
读取 Actions 日志
  ↓
定位失败 step
  ↓
判断错误类型
  ↓
读取相关配置文件
  ↓
生成修复建议
  ↓
用户确认后修改
  ↓
重新触发构建

这个场景很好，因为它有明确的验证信号：构建是否通过。

8.2 Text2SQL 自修复 Loop ​

text
理解用户问题
  ↓
检索 schema
  ↓
生成 SQL
  ↓
SQLGlot 校验
  ↓
执行 SQL
  ↓
如果报错，根据错误修 SQL
  ↓
返回结果和解释

这个场景也很适合做 Loop，因为 SQL 是否能执行、字段是否存在、结果是否为空，都可以检查。

8.3 技术博客生成 Loop ​

text
读取资料
  ↓
提取事实和观点
  ↓
生成大纲
  ↓
写初稿
  ↓
检查事实来源
  ↓
补代码和图
  ↓
输出文章包

你现在让我写博客，其实也可以被设计成一个 Loop：先找资料，再抽事实，再写大纲，再写文章，再检查引用，再打包。

9. Prompt 没死，只是被 Loop 包进去了 ​

有些文章会把这件事写成"Prompt 已死"。我觉得太绝对了。

Prompt 当然还重要。Loop 里的每一步，模型仍然需要 Prompt 才能理解上下文、选择工具、生成动作。变化在于：Prompt 不再一定由人手写，而是由系统根据当前状态自动生成。

过去是：

text
人写 Prompt → 模型回答

现在变成：

text
Loop 读取状态 → 自动生成 Prompt → 模型行动 → 系统验证 → 进入下一轮

所以更准确的说法是：

Prompt 没有消失，只是从"用户手写"变成了"系统生成"。

Boris 那句话真正值得注意的地方也在这里。他不是说自己不再使用 Prompt，而是说他不再直接写 Prompt。他写的是能持续生成 Prompt、调用 Claude、检查结果、决定下一步的 Loop。

10. 总结 ​

如果把这篇文章压缩成一句话，我会这么说：

Prompt 解决的是"这次怎么说"，Loop 解决的是"这件事怎么一步步做完"。

Boris Cherny 那句"我写 loop"之所以能传播，是因为它刚好说中了 AI 编程工具的阶段变化：当模型能力足够强，工具调用足够成熟，开发者真正要设计的就不只是提示词，而是一套可迭代、可验证、可停止、可观察的任务系统。

结合 2026 年这几次采访和 Anthropic 的产品动作来看，Agent Loop 不是一个孤立概念。它和 Claude Code、后台任务、Routines、多 agent 并行、MCP、IDE 集成、GitHub Actions 这些能力是一起出现的。

对普通开发者来说，现在最现实的路线不是一上来就追求"全自动软件工程师"，而是先把一个小任务做成 Loop：

  1. 有明确目标；
  2. 有状态；
  3. 有工具；
  4. 有验证；
  5. 有停止条件；
  6. 有日志；
  7. 有人工接管边界。

能稳定写出这些小 Loop，再回头看"我已经不写 prompt 了，我写 loop"，它就不是一句口号，而是一种真正的工程方法。
